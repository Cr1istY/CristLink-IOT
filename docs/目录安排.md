在 Go 语言项目中实现分布式服务，核心在于如何在一个代码仓库（Monorepo）中优雅地管理多个独立的服务，同时保持代码复用和清晰的边界。

根据 Go 社区广泛认可的最佳实践，推荐采用以下目录结构来组织你的项目。这种结构既能保证服务间的独立性，又能方便地共享公共代码。

### 🗂️ 推荐的分布式项目目录结构

一个典型的、适用于中大型分布式项目的结构如下：

```
myproject/
├── cmd/                      # 1. 应用程序入口
│   ├── user-service/         # 用户服务
│   │   └── main.go
│   ├── order-service/        # 订单服务
│   │   └── main.go
│   └── gateway/              # API网关
│       └── main.go
├── internal/                 # 2. 私有应用代码
│   ├── user/                 # 用户模块的私有代码
│   │   ├── handler/
│   │   ├── service/
│   │   └── repository/
│   ├── order/                # 订单模块的私有代码
│   │   ├── handler/
│   │   ├── service/
│   │   └── repository/
│   └── common/               # 项目内共享的私有代码
│       ├── middleware/
│       └── utils/
├── pkg/                      # 3. 公共库代码
│   ├── logger/               # 可被外部项目导入的日志库
│   └── response/             # 统一的API响应格式
├── api/                      # 4. API 定义
│   └── proto/                # Protocol Buffers 定义
│       ├── user.proto
│       └── order.proto
├── configs/                  # 5. 配置文件
│   ├── config.yaml
│   └── config.prod.yaml
├── scripts/                  # 6. 脚本文件
│   ├── build.sh
│   └── deploy.sh
├── go.mod                    # 7. 模块定义
├── go.sum
└── README.md
```

### 🧩 核心目录详解

这种结构的关键在于 `cmd`, `internal`, 和 `pkg` 这三个目录的分工：

1.  **`cmd/` (应用程序入口)**
    *   **作用**：存放所有可执行程序的入口文件。每个子目录代表一个独立的服务或工具，其名称通常与最终生成的二进制文件名一致。
    *   **原则**：`main.go` 文件应该非常简洁，主要负责初始化配置、连接数据库、调用 `internal` 中的服务启动逻辑等，避免在此处编写复杂的业务代码。

2.  **`internal/` (私有应用代码)**
    *   **作用**：这是 Go 语言特有的访问控制机制。放在 `internal` 目录下的代码，**只能被同一模块内的其他包导入，外部项目无法导入**。这为项目的内部实现提供了强大的封装性。
    *   **组织方式**：你可以按**业务领域**（如 `user/`, `order/`）来组织代码，将每个业务模块的 `handler`（处理器）、`service`（业务逻辑）、`repository`（数据访问）都放在对应的目录下，实现高内聚。同时，也可以放置 `common/` 目录，存放项目内部各服务共享的中间件、工具函数等。

3.  **`pkg/` (公共库代码)**
    *   **作用**：存放那些设计良好、足够通用、可以被**外部项目**导入和复用的库代码。
    *   **使用建议**：要谨慎使用 `pkg` 目录。只有当你确定某段代码（如一个精心封装的日志库、一个通用的响应工具）确实需要被其他项目引用时，才把它放在这里。对于大多数项目，将所有代码放在 `internal` 中是更简单、更安全的选择。

### ⚙️ 依赖管理

在分布式项目中，依赖管理至关重要。

*   **统一模块**：对于大多数情况，整个项目使用一个 `go.mod` 文件就足够了。所有服务都从这个根模块导入代码，例如 `import "myproject/internal/user/handler"`。
*   **多模块 (进阶)**：如果你的项目规模非常大，或者某些服务需要独立版本化，可以考虑在 `cmd/` 下的每个服务目录中创建独立的 `go.mod` 文件。在开发阶段，可以使用 `replace` 指令来指向本地的其他模块路径，方便联调。

### 🚀 构建与运行

使用这种结构，你可以非常方便地构建和运行不同的服务：

```bash
# 构建用户服务
go build -o bin/user-service ./cmd/user-service

# 构建订单服务
go build -o bin/order-service ./cmd/order-service

# 运行 API 网关
go run ./cmd/gateway
```

通过这种方式，你的项目结构清晰、职责明确，既支持了分布式部署，又保证了代码的可维护性和扩展性。
