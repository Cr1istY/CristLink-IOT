在 GoLink 平台中，如果说 `gateway` 模块是负责修路和通关的**“港口码头”**，那么**设备管理模块**就是负责资产登记、状态监控和调度指挥的**“港口管理局”**。

它位于**微服务业务逻辑层**，是物联网平台的核心资产中心。它不直接处理底层的 TCP 连接或二进制流，而是专注于**设备的全生命周期管理**和**数字孪生状态的维护**。

以下是该模块在系统中的 5 个核心作用详解：

### 1. 设备全生命周期档案（物模型与注册）
这是设备管理最基础的功能，相当于给每个物理设备在数字世界办理“身份证”和“户口本”。

*   **设备注册与鉴权**：
    *   负责设备的录入（手动录入或批量导入）。
    *   生成和分发设备凭证（如 `DeviceID`、`Secret`、X.509 证书）。
    *   **作用**：当 `gateway` 模块进行连接鉴权时，需要调用设备管理模块（或查询其缓存）来确认“这个设备是否存在”以及“密钥是否正确”。
*   **物模型管理**：
    *   定义设备的“能力”。比如定义一个“智能电表”有哪些属性（电压、电流）、有哪些服务（远程合闸）、有哪些事件（掉电报警）。
    *   **作用**：为上层应用提供标准化的数据描述，让应用层知道设备上报的 `0x01` 代表“开”，`0x00` 代表“关”。

### 2. 设备影子与状态同步
物理设备经常离线或网络不稳定，应用层不能直接去查设备的实时状态。设备管理模块通过维护**设备影子**来解决这个问题。

*   **状态缓存**：
    *   在 Redis 中维护一份设备的最新状态（如：`{ "switch": "on", "temp": 25 }`）。
    *   当 `gateway` 收到设备上报的数据（通过 Kafka）时，设备管理模块负责消费这些数据，并更新 Redis 中的影子。
*   **期望状态 vs 实际状态**：
    *   当用户想关灯但设备离线时，设备管理服务会在影子中记录“期望状态=关”。
    *   一旦设备上线，网关上报状态，设备管理服务对比发现不一致，会自动触发指令下发，将设备状态同步为用户期望的状态。
    *   **作用**：屏蔽了设备离线的复杂性，让应用层感觉设备永远是“在线且可控”的。

### 3. 远程控制与指令下发
这是设备管理的“指挥”功能，负责实现云端对设备的反向控制。

*   **指令封装与路由**：
    *   接收来自 API 或规则引擎的控制请求（如“打开阀门”）。
    *   根据设备 ID 找到设备所属的网关节点（如果网关是多实例集群）。
    *   将控制命令封装成设备能听懂的格式（JSON 或二进制），推送到 Kafka 的指令 Topic，或者通过 RPC 直接调用 `gateway` 模块。
*   **指令流转管理**：
    *   记录指令的发送时间、超时时间。
    *   监听设备的指令应答（通过 Kafka），判断指令是“执行成功”还是“执行失败”，并更新指令记录。

### 4. 遥测数据清洗与持久化
虽然 `gateway` 负责接收数据，但数据的“理解”和“存储”是由设备管理模块（或其配合的数据流模块）完成的。

*   **数据解析与标准化**：
    *   消费 Kafka 中的原始数据。
    *   根据物模型定义，将原始值转换为工程值（例如：将传感器上报的 `1024` 转换为 `25.5°C`）。
*   **数据存储**：
    *   **元数据**：将设备的静态信息（名称、型号、安装位置）存入 **MySQL**。
    *   **时序数据**：将设备的动态数据（温度曲线、电压波动）存入 **TDengine**。
    *   **作用**：为前端展示和历史数据查询提供数据支撑。

### 5. 运维与诊断
设备管理模块还是设备健康的“医生”。

*   **在线状态管理**：
    *   监听 `gateway` 发出的“设备上线/下线”事件。
    *   更新数据库中的 `last_seen_time` 和 `status` 字段。
*   **固件升级**：
    *   管理固件版本库。
    *   下发升级任务，记录设备的升级进度（如：下载中、升级中、升级成功）。
*   **日志管理**：
    *   收集设备上报的运行日志，供技术人员远程排查故障。

---

### 📌 总结：设备管理在数据流中的位置

你可以把**设备管理模块**想象成**“中央控制室”**：

1.  **档案室**：它记录着所有设备是谁、属于谁、有什么能力（**MySQL/物模型**）。
2.  **监控大屏**：它实时显示着所有设备的最新状态（**Redis/设备影子**）。
3.  **调度员**：它接收用户的指令，翻译后发给码头（**Gateway**），让码头转交给设备。
4.  **记录员**：它把设备汇报的历史数据整理归档（**TDengine**）。

**在你的代码中，`internal/modules/device` 应该包含：**
*   **API 层**：处理 HTTP/gRPC 请求（如 `CreateDevice`, `GetStatus`）。
*   **业务逻辑层**：处理状态机流转、影子比对、指令封装。
*   **数据访问层**：操作 MySQL（元数据）、Redis（影子）、TDengine（时序数据）。
*   **Kafka 消费者**：监听设备上报的数据和事件。

**它不应该包含：**
*   Socket 网络监听代码。
*   底层的二进制编解码逻辑（这些属于 `gateway`）。

这就是设备管理模块作为**“业务核心”**的纯粹职责。
