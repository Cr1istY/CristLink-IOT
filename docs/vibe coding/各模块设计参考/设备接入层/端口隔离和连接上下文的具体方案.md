将**方案 1（端口隔离）**与**方案 4（连接上下文）**结合，是构建企业级物联网网关的**最佳实践**。

这种结合的核心思想是：**“端口决定大方向，上下文锁定具体设备”**。

*   **方案 1（端口）**：负责粗粒度分类。比如 1883 端口进来的全是 MQTT 流量，502 端口进来的全是 Modbus 流量。
*   **方案 4（上下文）**：负责细粒度绑定。在连接建立初期（`OnOpen`），解析设备的“身份证”（如 ProductKey/DeviceKey），并将其绑定到当前连接上。后续的 `OnMessage` 不需要重复解析身份，直接从上下文取用。

以下是这种架构在 GoLink 中的具体代码实现：

### 🏗️ 核心架构设计

我们需要定义一个**连接会话结构体（Session）**，用来存储在 `gnet.Conn` 的上下文中。

```go
// internal/gateway/session.go

type Session struct {
    ProductKey string
    DeviceKey  string
    Protocol   string // 记录协议类型，如 "mqtt", "modbus"
    IsAuthed   bool   // 是否已通过鉴权
    // 可以扩展更多字段，如 TenantID, Token 等
}
```

### 🚀 主程序：利用端口启动不同的服务

在 `main.go` 中，我们根据不同的端口启动不同的 Server 实例。虽然代码看起来是两个 Server，但它们可以共享底层的业务逻辑（如 Kafka Producer）。

```go
// main.go

func main() {
    // 共享的 Kafka 生产者
    producer := kafka.NewProducer(...)

    // 1. 启动 MQTT 服务器 (监听 1883)
    // 这个 Server 知道自己是处理 MQTT 的
    go func() {
        mqttServer := &GatewayServer{
            Port:     1883,
            Protocol: "mqtt", // 注入协议标识
            Producer: producer,
        }
        log.Fatal(gnet.Run(mqttServer, "tcp://:1883"))
    }()

    // 2. 启动 Modbus 服务器 (监听 502)
    // 这个 Server 知道自己是处理 Modbus 的
    go func() {
        modbusServer := &GatewayServer{
            Port:     502,
            Protocol: "modbus",
            Producer: producer,
        }
        log.Fatal(gnet.Run(modbusServer, "tcp://:502"))
    }()

    select {}
}
```

### ⚙️ 核心逻辑：OnOpen 与 OnMessage 的配合

这是结合方案 1 和方案 4 的精髓所在。

#### 1. OnOpen：连接建立时的“握手”与“注册”

当设备连接上来时，`gnet` 会调用 `OnOpen`。此时，我们结合 **端口信息**（已知是 MQTT）和 **握手数据**（设备 ID），初始化 Session。

```go
func (gs *GatewayServer) OnOpen(c gnet.Conn) (out []byte, action gnet.Action) {
    // 1. 创建一个新的 Session
    sess := &Session{
        Protocol: gs.Protocol, // 从 Server 注入的协议类型 (e.g., "mqtt")
        IsAuthed: false,
    }

    // 2. 对于 MQTT，通常需要等待 CONNECT 包
    // 注意：gnet 的 OnOpen 触发时，数据可能还没到。
    // 在实际 MQTT 实现中，我们通常在收到第一条数据（CONNECT 包）解析成功后，
    // 才真正完成“注册”。
    // 这里为了演示逻辑，假设我们先存一个空的 Session。
    
    // 3. 将 Session 存入 gnet 的上下文
    c.SetContext(sess)

    log.Printf("新连接建立: %s, 协议: %s", c.RemoteAddr().String(), gs.Protocol)
    
    // 返回 nil, None 表示保持连接打开，等待数据
    return nil, gnet.None
}
```

#### 2. OnMessage：利用上下文“无感”处理

当数据到来时，我们不再需要判断“这是什么协议”（因为端口已经确定了），也不需要重新解析“这是哪个设备”（因为 Session 里可能有，或者这是第一条包需要解析）。

**针对 MQTT 的特殊处理逻辑：**
MQTT 的第一包必须是 `CONNECT`。我们需要在 `OnMessage` 里做一个状态机判断。

```go
func (gs *GatewayServer) OnMessage(c gnet.Conn, msg []byte) (out []byte, action gnet.Action) {
    // 1. 从上下文获取 Session
    sess, ok := c.Context().(*Session)
    if !ok {
        log.Println("上下文错误")
        return nil, gnet.Close
    }

    // 2. 根据协议类型处理（虽然端口已经区分了，但为了代码复用，这里可以 switch）
    switch gs.Protocol {
    case "mqtt":
        return gs.handleMQTT(c, sess, msg)
    case "modbus":
        return gs.handleModbus(c, sess, msg)
    }
    return nil, gnet.None
}

// --- MQTT 具体处理器 ---
func (gs *GatewayServer) handleMQTT(c gnet.Conn, sess *Session, msg []byte) gnet.Action {
    // 场景 A: 还没鉴权，收到的必须是 CONNECT 包
    if !sess.IsAuthed {
        // 1. 解析 MQTT CONNECT 包 (伪代码)
        // 这里需要解析二进制，提取 ClientID, Username(作为DeviceKey)
        clientID, deviceKey, err := parseMQTTConnect(msg)
        if err != nil {
            log.Println("MQTT 握手失败")
            return gnet.Close
        }

        // 2. 更新 Session 信息
        sess.DeviceKey = deviceKey
        sess.ProductKey = "pk_from_config" // 实际应从 Username 或端口配置解析
        sess.IsAuthed = true
        
        // 3. 更新回上下文
        c.SetContext(sess)
        
        log.Printf("设备 %s 鉴权成功", sess.DeviceKey)
        // 这里通常需要回复 CONNACK 包给设备
        return gnet.None
    }

    // 场景 B: 已鉴权，处理正常上报数据
    // 此时 sess.DeviceKey 已经有了，直接复用！
    
    // 1. 获取解析器
    codec, _ := protocol.GetCodec("mqtt")
    
    // 2. 准备元数据 (从 Session 取，而不是重新解析)
    meta := map[string]string{
        "product_key": sess.ProductKey,
        "device_id":   sess.DeviceKey,
        // Topic 需要从 MQTT 包中解析出来，这里简化处理
        "topic":       parseTopicFromMQTTPacket(msg), 
    }

    // 3. 解码并发送 Kafka
    payload, err := codec.Decode(msg, meta)
    if err != nil {
        return gnet.Close
    }
    
    go gs.sendToKafka(payload)
    return gnet.None
}
```

### 📌 为什么要这样结合？（优势分析）

1.  **性能极致（方案 1 的贡献）**
    *   你不需要在 `OnMessage` 里写 `if bytes.HasPrefix(msg, []byte("POST"))` 这种代码来猜协议。
    *   代码路径最短：端口 -> 专用 Handler。

2.  **状态保持（方案 4 的贡献）**
    *   **避免重复解析**：设备 ID 通常在第一条包（CONNECT）里。解析一次存入 `Session`，后续 1000 条数据上报都不需要再动字符串，直接取 `sess.DeviceKey`。
    *   **连接级鉴权**：如果设备没通过 `OnOpen` 或第一条包的校验，直接 `gnet.Close`，后续任何数据都会被拒绝，非常安全。

3.  **代码清晰**
    *   `main.go` 负责网络拓扑（端口）。
    *   `session.go` 负责状态管理。
    *   `handler.go` 负责业务逻辑。

这就是 GoLink 这种高性能网关处理海量设备连接的标准姿势。
